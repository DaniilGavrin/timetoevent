## 🚀 Начало работы

### 1. Форк репозитория

Нажми кнопку **Fork** на GitHub или:

```bash
git clone https://github.com/DaniilGavrin/TimeToEvent.git
cd TimeToEvent
```

### 2. Настройка окружения

Подробная инструкция — в [`INSTALL.md`](./INSTALL.md).

**Минимум:**
- Node.js 20+
- Rust 1.75+ (через rustup)
- Visual Studio Build Tools 2022 (Windows) или системные пакеты (Linux)

### 3. Установка зависимостей

```bash
npm install
```

### 4. Запуск в режиме разработки

```bash
npm run tauri dev
```

Первый запуск займёт 5–15 минут (компиляция ~670 Rust-зависимостей).

### 5. Создание ветки

```bash
git checkout -b feature/your-feature-name
# или
git checkout -b fix/your-bug-fix
```

## 💡 Принципы разработки

1. **MVP first** — сначала рабочее ядро, потом фичи
2. **Local-first** — всё работает офлайн, синхронизация опциональна
3. **Privacy by design** — шифрование по умолчанию, без серверов
4. **Simple > Complex** — если можно сделать проще — делаем проще
5. **Test everything** — каждый модуль с тестами
6. **Document as you go** — документация параллельно с кодом
7. **No iOS/macOS** — осознанное ограничение, фокус на Windows/Linux/Android
8. **KISS** — Keep It Simple, Stupid
9. **YAGNI** — You Aren't Gonna Need It
10. **DRY** — Don't Repeat Yourself

## 🎨 Стиль кода

### Rust

- **Форматирование:** `rustfmt` (настройки по умолчанию)
- **Линтинг:** `clippy` (все warnings должны быть исправлены)
- **Именование:** snake_case для функций/переменных, CamelCase для типов
- **Документация:** rustdoc для публичных API (`///`)
- **Обработка ошибок:** `Result<T, String>` для Tauri-команд, конкретные ошибки для внутренних функций

```bash
# Форматирование
cargo fmt

# Линтинг
cargo clippy --all-targets --all-features -- -D warnings

# Тесты
cargo test
```

### TypeScript / React

- **Форматирование:** Prettier (настройки по умолчанию)
- **Линтинг:** ESLint (если настроен)
- **Типизация:** строгая (`strict: true` в tsconfig.json)
- **Компоненты:** функциональные с хуками (никаких class-компонентов)
- **Стейт:** Zustand для глобального, `useState` для локального
- **Стили:** Tailwind CSS (никаких inline-стилей, кроме динамических значений)

```bash
# Форматирование
npx prettier --write "src/**/*.{ts,tsx}"

# Проверка типов
npx tsc --noEmit
```

### Общие правила

- **Длина строки:** максимум 100 символов
- **Имена переменных:** осмысленные, не `x`, `y`, `tmp` (кроме циклов)
- **Комментарии:** объясняют **почему**, а не **что**
- **Dead code:** не коммитить закомментированный код
- **Console.log:** не коммитить отладочные выводы (используй `log::info!` в Rust)

## 📝 Коммиты

Используем **Conventional Commits** формат:

```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

### Типы

| Тип | Описание |
|-----|----------|
| `feat` | Новая фича |
| `fix` | Исправление бага |
| `docs` | Изменения в документации |
| `style` | Форматирование, точки с запятой, etc. |
| `refactor` | Рефакторинг без изменения функциональности |
| `test` | Добавление/изменение тестов |
| `chore` | Изменения в сборке, зависимостях, CI |
| `perf` | Улучшение производительности |
| `ci` | Изменения в CI/CD |

### Примеры

```
feat(events): добавить форму создания события
fix(sync): исправить race condition в broadcast
docs(readme): обновить инструкции по установке
refactor(crypto): вынести HKDF в отдельную функцию
test(aes): добавить тесты для шифрования/расшифровки
chore(deps): обновить tokio до 1.52.3
```

### Правила

- **Один коммит = одно логическое изменение**
- **Заголовок:** максимум 72 символа, начинается с заглавной буквы, без точки в конце
- **Тело:** объясняет **что** и **почему** (не **как** — это видно в коде)
- **Не коммить** `node_modules/`, `target/`, `.env`, ключи

##  Pull Request'ы

### Перед созданием PR

1. **Обнови ветку** с `main`:
   ```bash
   git fetch origin
   git rebase origin/main
   ```

2. **Прогони тесты**:
   ```bash
   # Rust
   cargo test
   cargo clippy --all-targets --all-features -- -D warnings
   
   # TypeScript
   npx tsc --noEmit
   ```

3. **Проверь форматирование**:
   ```bash
   cargo fmt
   npx prettier --write "src/**/*.{ts,tsx}"
   ```

4. **Обнови документацию** (если нужно):
   - `CHANGELOG.md` — если изменилась функциональность
   - `README.md` — если изменилась установка/использование
   - `ARCHITECTURE.md` — если изменилась архитектура

### Создание PR

1. Запушь ветку:
   ```bash
   git push origin feature/your-feature-name
   ```

2. Создай PR на GitHub с шаблоном:

```markdown
## Описание
Что делает этот PR?

## Изменения
- Изменение 1
- Изменение 2

## Тестирование
Как ты тестировал изменения?

## Скриншоты (если применимо)
[скриншоты]

## Чеклист
- [ ] Код отформатирован
- [ ] Тесты проходят
- [ ] Документация обновлена
- [ ] Нет dead code
```

### Review

- PR должен быть reviewed хотя бы одним человеком (или автором через 24 часа для solo-проекта)
- Все комментарии должны быть resolved перед merge
- CI должен пройти (если настроен)

## 🧪 Тестирование

### Rust

```bash
# Все тесты
cargo test

# Тесты конкретного модуля
cargo test --lib crypto

# Тесты с выводом
cargo test -- --nocapture

# Integration tests
cargo test --test integration
```

**Правила:**
- Каждый публичный модуль должен иметь unit-тесты
- Тесты должны быть независимыми (не зависеть от порядка запуска)
- Моки для внешних зависимостей (БД, сеть)
- Покрытие: минимум 70% для критичных модулей (crypto, sync)

### TypeScript

```bash
# Запуск тестов (когда настроим Vitest)
npm run test

# Coverage
npm run test:coverage
```

**Правила:**
- Тесты для Zustand stores
- Тесты для хуков (useTimer, useEvents)
- Тесты для утилит (dateUtils, formatters)
- Snapshot-тесты для UI-компонентов

### Ручное тестирование

Перед PR проверь:
- [ ] Приложение запускается (`npm run tauri dev`)
- [ ] Сборка проходит (`npm run tauri build`)
- [ ] Нет регрессий в существующем функционале
- [ ] Новые фичи работают как ожидается

## 📚 Документация

### Когда обновлять документацию

- **`CHANGELOG.md`** — при любом изменении функциональности
- **`README.md`** — при изменении установки/использования
- **`ARCHITECTURE.md`** — при изменении архитектуры
- **`API.md`** — при добавлении/изменении Tauri-команд
- **`USAGE.md`** — при изменении UI/UX
- **`FAQ.md`** — при добавлении частых вопросов
- **`TROUBLESHOOTING.md`** — при решении новых проблем

### Стиль документации

- **Язык:** русский (основной), английский (опционально)
- **Формат:** Markdown
- **Код:** всегда в code blocks с указанием языка
- **Ссылки:** относительные (`./INSTALL.md`, не `docs/INSTALL.md`)
- **Изображения:** в папке `docs/images/`

## 🤝 Code of Conduct

- **Уважение** — все участники заслуживают уважения
